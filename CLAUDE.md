# line-coach-bot（HOZYO）について

コーチと生徒（部活・クラブ）をマッチングするLINEボット（Node.js/Express）。個人向け（コーチ⇔生徒）とクラブ向け（クラブ登録・依頼）の2系統のフローを持つ。

## 技術スタック

- Express + `@line/bot-sdk`（個人用チャネルとクラブ用チャネルの2チャネル構成、`src/lineClient.js` / `src/clubClient.js`）
- データストアはGoogle Apps Script（GAS）経由のスプレッドシート（`src/gasClient.js`、`gas/SheetAPI.gs`）
- デプロイ先: Render（`.env.example` の `RENDER_URL`）

## ディレクトリ構成

- `src/index.js` — エントリポイント・ルーティング
- `src/flow.js` / `src/clubFlow.js` — 個人／クラブそれぞれの会話フロー（登録・マッチング・シフト・欠席報告・評価）
- `src/admin.js` / `src/clubAdmin.js` — 管理者向けPostback処理
- `src/certificate.js` — 認定証発行
- `scripts/` — リッチメニュー設定用スクリプト

# 行動ルール（恒久）

## 1. LINE IDは管理者以外に一切表示しない

LINE IDは個人情報。生徒・コーチ向けのチャット、通知メッセージのいずれにも、他ユーザーのLINE IDを直接埋め込まない。氏名表示が必要な場合はGAS側で解決してから渡す。実装前に「非管理者にLINE IDが見える経路がないか」を必ず確認する。

（参考: [[feedback_line_id_privacy]] — okuri-botで実際に発生した露出パターンを踏まえたルール。同種のLINEマッチングボットである本プロジェクトにも適用）

## 2. 新機能は詳細を議論してから実装する

「実装しましょう」という承認が出ても、仕様に曖昧な点（対象者・表示形式・データ項目の有無など）が残っている場合は、着手前に質問して確認する。単純な修正は対象外。

## 3. 積極的な最適化提案を歓迎

修正・実装の過程で気づいた別の改善点（潜在バグ、設計上の弱点、将来のリスクなど）は、聞かれなくても提案してよい。ただし提案後、実装に進む前に方針を確認する（ルール2）。
